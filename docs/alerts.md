# Grafana alert-rule pack

[`deploy/grafana/alerts.yaml`](../deploy/grafana/alerts.yaml) is a Grafana **alerting file
provisioning** manifest (generated by
[`deploy/grafana/gen_alerts.py`](../deploy/grafana/gen_alerts.py) — never hand-edit; regenerate
with `just gen-grafana`). It ships 14 rules across 4 severity groups in a "sf2loki" folder, covering
every data-loss and degradation signal the connector's metrics expose.

## Provisioning

Drop `alerts.yaml` into Grafana's alerting provisioning directory
(`<paths.provisioning>/alerting/`) so Grafana picks it up on the next reload/restart — or push it
via the HTTP provisioning API. The generated file's `datasourceUid` is a placeholder
(`${DS_PROMETHEUS}`, matching the dashboard's own datasource input token): either regenerate with
`SF2LOKI_PROMETHEUS_DS_UID=<real-uid> python3 deploy/grafana/gen_alerts.py`, or `envsubst`/`sed`
the placeholder before deploying, so it points at your Prometheus-compatible datasource's real UID.

Only `severity` labels are set (`critical` / `warning` / `info`) — notification routing is left to
your contact points and notification policies.

## Shipped alerts

| Alert | Severity | Meaning | First response |
|---|---|---|---|
| EntriesDropped | critical | Log entries permanently dropped as undeliverable (bad_request / oversized_413) — real, permanent data loss. | Check the dashboard's "Entries dropped rate by reason" panel for the reason label, then check Loki-side limits (out-of-order window, max entry size) or reduce `max_line_bytes` if oversized. |
| ReplayFallback | critical | A Pub/Sub subscription restarted from a fallback replay position after an invalid/expired replay_id — events between the last commit and the fallback point may have been skipped. | Identify the affected topic and, if the gap matters, backfill via SOQL/EventLogFile for that topic and time window. |
| PushesStalled | critical | No successful Loki push in 15+ minutes while the internal queue is non-empty — the sink is failing, not idle. | Check Loki connectivity/auth (`tenant_id`, `auth_token`) and Loki-side health; if Loki looks healthy, check for a stuck push (network partition, DNS). |
| StreamDown | critical | A Pub/Sub topic's subscribe stream has been down for 15+ minutes — real-time events for that topic aren't flowing at all. | Check connector logs for that topic's reconnect/auth errors and Salesforce Pub/Sub API status; restart the connector if it isn't self-recovering. |
| WatermarkStale | critical | A source/object's SOQL polling watermark hasn't advanced in 4+ hours (4x a typical 1h poll interval) — that object's ingestion is effectively stuck. | Check "SOQL poll errors rate" for that source/object and connector logs; verify the SOQL query/permissions are still valid. |
| SoqlPollErrors | warning | A source/object's SOQL poll cycles are failing — that object's watermark isn't advancing. | Check connector logs for the SOQL error detail (auth, malformed query, permissions) for the affected source/object. |
| StreamStalls | warning | The Pub/Sub keepalive watchdog force-reconnected a stream — usually self-recovers but signals instability. | If this recurs for the same topic, check network stability and reconnect/backoff configuration. |
| ApiThrottled | warning | Salesforce REST calls are being rejected with `REQUEST_LIMIT_EXCEEDED`. | Check "API usage % used by limit" on the dashboard; raise poll intervals or the org's API limit allocation. |
| ApiBudgetLow | warning | Less than 10% of the `DailyApiRequests` org limit remains. | Check the org limits dashboard row for which pollers are consuming budget; reduce poll frequency/scope until the daily limit resets. |
| EgressPaused | warning | The daily egress byte budget is exhausted — pushes to Loki are paused until the next UTC day. | Check "Egress budget used bytes" and the configured daily budget; raise the budget or reduce ingestion scope (sampling, drop_row rules). |
| LeadershipAnomaly | warning | `sum(sf2loki_leader) != 1` for 10m+ — either no replica is ingesting (leaderless) or two are (split-brain). | Sustained 0: check lease-file storage reachability and replica logs. Sustained 2+: verify both replicas point at the SAME lease file on shared storage and hosts are NTP-synced. |
| IngestLagP95High | warning | p95 ingest lag has exceeded 6h — above normal EventLogFile batch latency (3–6h). | Check the ingest-lag panel by `event_type` to isolate which path is lagging, then check queue depth / Loki push health for backpressure. |
| TimestampFallbacks | info | Entries are using a fallback timestamp (e.g. ingest time) instead of the true Salesforce event time — a fidelity issue, not data loss. | Check "Timestamp fallbacks rate" by source; check whether Salesforce is returning malformed date fields for that object. |
| LinesTruncated | info | Log lines are being truncated to `max_line_bytes` before push — a fidelity issue, not data loss. | Check "Lines truncated rate" by source; raise `max_line_bytes` for that source if full payloads are required. |

Thresholds live as named constants at the top of `gen_alerts.py` with rationale comments — tune
them there and regenerate rather than editing the YAML.
